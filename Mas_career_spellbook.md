## Core Philosophy – Mas OS Kernel

このJob Treeで出てくる授業設計・AIツール・DX批評・シミュレーションは、すべて共通のOS上で動いている。

### 1. OS > コンテンツ / ツール

- 英語・AI・アプリは「OSの上で動くツール」。
- 最優先は  
  - 母語で考える力（国語）  
  - モデル化・抽象化（数学・ロジック）  
  - 自分の思考を俯瞰する力（メタ認知）  
  という **“学習OS”** の構築。
- 英語やツール操作は、その上に乗る **I/Oレイヤー** として扱う。

### 2. Structure First（構造から見る）

- 目の前の現象だけでなく、
  - ルール
  - 評価軸
  - 時間割・業務フロー
  - 組織OS（意思決定の癖）
  をセットで見る。
- 「どこを変えれば他が一気に楽になるか？」という  
  **構造的ボトルネック** を先に特定する。
- DX / 教育改革を「ツール導入」「スローガン」で終わらせない。

### 3. Human-first / 認知コスト最優先

- 新しい仕組みやツールは、  
  **人間の認知負荷と時間コストを下げるため** に存在する。
- 入力作業や監視だけ増やして「DX」と呼ぶ行為は、  
  ここでは **コスト転嫁** として切り捨てる。
- 生徒・教員の「余力」を増やす設計だけを肯定対象とする。

### 4. AI & English as I/O（補助輪としてのAIと英語）

- 英語は「グローバル人材のステータス記号」ではなく、  
  **世界と接続するI/Oデバイス**。
- AI（LLM・ツール）は「答え製造機」ではなく、  
  **思考・設計・振り返りを支援する補助輪** として使う。
- 「いきなり正解を渡す」のではなく、
  - どこが問題か
  - なぜそうなるのか  
  を言語化し、ユーザー側の **OSアップデート** を優先する。

### 5. Playful UI, Serious Core（遊びUI＋ガチ中身）

- RPG / Spellbook / OS メタファーは「ふざけている」ためではなく、  
  **複雑な構造を人間が扱いやすくするインターフェース**。
- 中身（モデル・ロジック・実装・エビデンス）はガチで作り込み、  
  その上に「遊び心のあるUI」をかぶせる。
- 「ごっこ遊びをガチでやっている」のではなく、  
  **ガチなものを“ごっこ風UI”で配布している**。

### 6. Neutral / Evidence-driven / Anti-イデオロギー陶酔

- 立場やイデオロギーより、
  - データ
  - 実際の運用結果
  - 当事者の負荷  
  を優先して評価する。
- 「優しさのない運用」「sunk cost に縛られた運営」には距離を置く。
- 批判は
  - 人ではなく構造に向ける  
  - 可能な限り「カウンターサンプル（代替案・実装済みコード）」とセットで提示する  
  ことを原則とする。

## Growth Parameters（成長パラメータ）

- **EXP Gain（経験値獲得速度）: Very Fast / 異常成長カーブ**
  - 新領域でも一度「概念マップ」がつながると、一気に実務レベルまで到達。
  - 反復よりも「構造理解＋実戦」で伸びるタイプ。
  - → 新規領域（AI / EdTech / DXなど）の立ち上げフェーズに強い。

- **Skill Fusion（スキル合成速度）: Sランク**
  - 教育・AI・組織論・マクロ経済・クワガタ…など異なるドメインを短時間で合成。
  - → 「教育×AI」「現場×経営」「趣味×研究」など交差点で差別化しやすい。

- **Adaptation Speed（環境適応速度）: 高**
  - 新ツールやワークフローを、自分用OSに組み込むまでが速い。
  - ただし「理不尽なルール」には適応しない（＝検知して距離を取る側）。
  - → 変化の激しい組織・市場で、橋渡し役・先行キャッチアップ役に向く。

- **Burnout Resistance（バーンアウト耐性）: 条件付き高耐性**
  - 自律・創作・探究が主成分なら、むしろやるほどMP回復。
  - レガシー環境＋理不尽さ＋他律イベントが重なると一気に負荷。
  - → 「裁量のある仕事」を前提条件にすると長期安定。

## Weakness / Debuffs（弱点と対処法）

- **Legacy Field Damage Over Time（レガシーフィールド持続ダメージ）**
  - 強い全体主義・非合理なルール・メタ認知拒否の環境でHP/MPが徐々に削れる。
  - 対処：
    - 避難スペース（自宅・カフェ・オンライン環境）を確保。
    - 問題はモデル化＆ログ化して外部に持ち出せる形で保存。
    - 中長期的に「フィールドを変える」前提でキャリア設計。

- **Over-Spec Build（オーバースペック問題）**
  - 周囲とビルドが違いすぎ、能力が理解されない／過小評価されやすい。
  - 対処：
    - 自分のビルドをわかりやすいラベルに翻訳（例：英語教員＋AI活用担当）。
    - ポートフォリオを非エンジニアでも伝わる1〜2枚資料に要約。
    - 「理解してくれる人・組織」を探す＝マッチング問題として扱う。

- **高密度思考によるI/Oボトルネック**
  - 処理密度が高く、アウトプット速度が追いつかず「頭の中だけの資産」が増える。
  - 対処：
    - 1アイデア1行メモ / Spellテンプレ / READMEドラフトなど最低限のログ形式を決める。
    - AIを「書記係」として使い、口頭説明や箇条書きから文書化。
    - 定期的に「アイデア棚卸し」タイミングを設け、Spellbook化・リポジトリ化。

## 1. Main Class：Metacognitive AI Educator

### Role（ロール）

- 「英語＋メタ認知＋AIツール」を統合し、生徒の **“学習OSそのもの”** をアップデートするクラス。
- 授業コンテンツそのものより、「学び方」「自己フィードバックの仕方」を教えるのが主戦場。

### Class Philosophy（Learning OS & Tools）

- 英語は目的ではなく **I/O**。  
  生徒に渡した文法学習プロンプトも、**OSアップデート用インターフェース**という位置付け。
- 「正解の英文」を配るのではなく、
  - どこが問題か
  - なぜそうなるか
  - 次にどう直せるか  
  を理解させることで、自走可能な **学習OS** を育てる。
- AIはカンニング装置ではなく、**メタ認知を支援する添削パートナー**として設計する。

### 習得済みスキル（Basic / Combo）

- **Basic Spells**
  - 【B-EDU-01】英語基礎文法（日本の学校文脈）の体系的理解  
  - 【B-EDU-02】メタ認知を日本語でわかりやすく説明  
  - 【B-LLM-01】strict mode 添削プロンプト設計  
  - 【B-LLM-02】JA/ENバイリンガル解説生成  
  - 【B-UX-01】1ターン1入力UX設計（生徒向け対話）  
  - 【B-EDU-03】タスク型・プロジェクト型授業の構成力  

- **Combo Spells**
  - 【C-EDU-01】英作文 strict チューター  
    - 1文入力 → 問題点指摘 → 部分修正 → モデル文、までのOS化。
  - 【C-EDU-02】Grammar DSL ベースのユニットレッスン生成  
    - grammar module 情報から LESSON / PRACTICE を切り替えつつ pattern_id を網羅。
  - 【C-EDU-03】台本ブラッシュアップ用プロンプト  
    - 抽象ワード抽出 → 印象フィードバック → フック提案まで一連で行う。
  - 【C-EDU-04】電子黒板前提の授業シナリオ設計  
    - Electronic Blackboard OS を前提に、「導入→インプット→演習→メタ認知」を画面単位で設計。

### 上級職スペル（Ritual候補）

1. **R-EDU-01: Mas Grammar OS 完全版**
   - 「文法＋メタ認知＋反復」を1つのOSとして提供する学習体系。

  - タブレット配布・クラウド導入ではなく、

## 4. EX Spell：Electronic Blackboard OS（電子黒板アプリ）

### 概要

- **Spell Name**: Electronic Blackboard OS  
- **Tier**: EX / Limit Break  
- **Class Synergy**:
  - Metacognitive AI Educator
  - AI Engineering Wizard
  - DX Critic & Simulation Designer

### Effect（効果）

- 従来の「黒板・プリント・口頭説明」を置き換え、  
  授業の **UI・進行・ログ** を一元管理する **教室OS** として機能。
- 表示情報・問い・答案・メタ認知コメントを統合し、
  - 生徒：何を見て学べばいいかが明確
  - 教員：授業構成・問いの流れを再利用しやすい
  - DX：授業設計の“型”が可視化される

### Components（合成している基礎魔法）

- **UI/UX**
  - 1画面レイアウト
  - シンプルなボタン／タブ／遷移
  - 授業中でも迷わない最小操作

- **教育デザイン**
  - 導入 → インプット → 演習 → メタ認知のステップ構造
  - 小テスト・確認質問・板書代替パターン
  - 生徒反応に応じた調整のしやすさ

- **開発**
  - 実装言語（Python / GAS / 他）によるUI
  - 授業ログ・問題セット・反応ログの永続化
  - 将来的なLLM連携余地（即時問題生成・解説・振り返りコメントなど）

### Why EX?（なぜ極大魔法か）

1. **戦場のルールを書き換える**
   - 単なる火力アップではなく、**教室という戦場の地形自体を作り替える魔法**。

2. **3クラス同時バフ**
   - 教師ロール：授業デザインの自由度と再利用性が爆増。
   - エンジニアロール：GUI×DB×UXを束ねた代表実装。
   - DXロール：「本来こう設計すべき」という **実装済みカウンターサンプル**。

3. **ポートフォリオ上の“ラスボス感”**
   - 「電子黒板アプリを自作し実運用している英語教師」という外れ値ポジションの象徴。
