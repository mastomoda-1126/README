## 1. Core Philosophy – Mas OS Kernel

このJob Treeに出てくる授業設計・AIツール・DX批評・シミュレーションは、すべて共通のOS上で動いている。

### 1.1 OS > コンテンツ / ツール

- 英語・AI・アプリは **学習OSの上で動くツール**。
- 最優先で育てたいのは：
  - 母語で考える力（国語）
  - モデル化・抽象化（数学・ロジック）
  - 自分の思考を俯瞰するメタ認知  
  → これを **“学習OS”** と定義。

### 1.2 Structure First

- 現象ではなく **構造** を見る：
  - ルール・評価軸・時間割・業務フロー・組織OS（意思決定の癖）
- 「どこを変えれば他が一気に楽になるか？」という  
  **構造的ボトルネック** を先に特定する。

### 1.3 Human-first / 認知コスト最優先

- 新しい仕組みやツールは  
  **人間の認知負荷と時間コストを下げるため** に存在。
- 入力作業と責任だけ増やす「DX」は、ここでは **コスト転嫁** とみなす。
- 生徒・教員の「余力」が増える設計だけを肯定対象にする。

### 1.4 AI & English as I/O

- 英語は「グローバル人材バッジ」ではなく、**世界と接続するI/Oデバイス**。
- AI（LLM等）は「答え製造機」ではなく、**思考・設計・振り返りを支援する補助輪**。
- 「いきなり正解」ではなく、  
  **どこが問題か／なぜそうなるか** を言語化し、ユーザーのOSを更新する。

### 1.5 Playful UI, Serious Core

- RPG / Spellbook / OS メタファーは  
  **複雑な構造を人間が扱いやすくするUI** として採用。
- 中身（モデル・コード・エビデンス）はガチで作り込み、  
  その上に「遊び心のあるラッパー」を被せる。

### 1.6 Neutral & Evidence-driven

- 立場よりも：
  - データ
  - 運用結果
  - 当事者の負荷  
  を優先して評価。
- 批判は **人ではなく構造に向ける**。  
  可能な限り **代替案・実装済みコードとセット** で提示。

## 2. Parameters & Abilities

### 2.1 Growth Parameters

- **EXP Gain**: Very Fast  
  - 構造がつながると一気に実務レベルまで伸びる。
- **Skill Fusion**: S  
  - 教育・AI・組織論・マクロ経済・趣味（クワガタなど）を短時間で合成。
- **Adaptation Speed**: 高  
  - 新ツール・ワークフローの取り込みは速いが、理不尽には「適応しない」。
- **Burnout Resistance**: 条件付き高  
  - 自律・創作・探究が主成分ならMP回復。  
  - レガシー＋他律強制イベントが続くと急激に消耗。

### 2.2 Active Skills

- **Concept Fusion Cast**  
  異なる領域のアイデアをまとめてAIやコードに投げ込み、合成プロダクトや授業OSにする。

- **Emotion → Product Transmute**  
  怒り・違和感をスクリプト・シミュレーション・文章・授業案に変換。

- **Metacognitive Lens**  
  自分・他人・組織のズレを構造として言語化。

- **Rapid Prompt Forge**  
  「やりたいことの説明」から最小ターンで実用プロンプト／メタプロンプトを生成。

### 2.3 Passive Skills

- **Legacy OS Detection**  
  振る舞い・制度・時間割などから「どの時代のOSか」を直感的に検知。

- **Spellbook Serialization**  
  作ったものを README / DSL / テンプレ / プロンプトとして再利用可能な形で残す癖。

- **Asynchronous Thinking Loop**  
  別の作業中にも課題をバックグラウンド処理し、後から合成魔法として出てくる。

### 2.4 Debuffs & Counters

- **Legacy Field DOT**  
  全体主義・非合理・メタ認知拒否の環境で継続ダメージ。  
  → 避難スペース確保、現場の問題をモデル化・ログ化し、フィールド変更前提でキャリア設計。

- **Over-Spec Build**  
  能力が理解されず過小評価されがち。  
  → ラベル翻訳（例：英語教員＋AI活用担当）、非エンジニア向け1〜2枚サマリ、マッチング重視。

- **I/O Bottleneck**  
  頭の中の処理量にアウトプットが追いつかない。  
  → 1アイデア1行メモ＋AI書記係＋定期的なアイデア棚卸し。

## 3. Main Class：Metacognitive AI Educator

### 3.1 Role

- 「英語＋メタ認知＋AI」を統合し、生徒の **学習OSそのもの** をアップデートするクラス。
- コンテンツよりも「学び方・自己フィードバック」が主戦場。

### 3.2 Class Philosophy

- 英語は目的ではなく **I/O**。  
- 文法学習プロンプトや strict 添削は、  
  **正解配布ではなく OS アップデートインターフェース** として設計。
- AIはカンニング装置ではなく、メタ認知を支援する添削パートナー。

### 3.3 Key Spells

- **Basic**
  - 英語基礎文法（日本の学校文脈）の体系理解  
  - メタ認知の日本語解説  
  - strict mode 添削プロンプト設計  
  - JA/ENバイリンガル解説生成  
  - 生徒向け 1ターン1入力UX設計  
  - タスク型・プロジェクト型授業構成  

- **Combo**
  - **英作文 strict チューター**  
    1文入力 → 問題点指摘 → 部分修正 → モデル文。
  - **Grammar DSL ユニットレッスン**  
    LESSON / PRACTICE を切り替えつつ pattern_id を網羅。
  - **台本ブラッシュアッププロンプト**  
    抽象語抽出 → 印象フィードバック → フック語提案。
  - **電子黒板前提授業シナリオ**  
    Electronic Blackboard OS 上で「導入→インプット→演習→メタ認知」をデザイン。

### 3.4 Ritual Candidates

- **Mas Grammar OS 完全版**  
  文法＋メタ認知＋反復を一つの学習OSとしてパッケージ。

- **AI×メタ認知 授業パッケージ**  
  スクリプト＋プロンプト＋ワークシート＋シナリオの授業セット。

## 5. Sub Class 2：DX Critic & Simulation Designer

### 5.1 Role

- 学校という“保護されたエコシステム”をモデル化し、  
  レガシーOS vs 高適応OS の対立や時間割構造の歪みを可視化するクラス。

### 5.2 Class Philosophy（DX = 構造改革）

- 本気のDXは「ツール配布」ではなく、  
  **意思決定ルール・評価軸・労働時間設計・業務フロー前提** に踏み込むことから始まる。
- 現場に入力作業と責任だけ増やすものは**DXではなくコスト転嫁**。
- 学校DXでは「タブレット導入」ではなく、  
  **教員の時間・生徒の学び・労働構造がどう変わったか** を最初に問う。

### 5.3 Key Spells

- **Basic**
  - 組織構造・インセンティブ設計の直感理解
  - Pythonシミュレーション
  - 辛口だが論理的な文章（note等）
  - マクロ経済・流動性・中央銀行メタファー
  - 労働時間・時間割のコストモデル化  

- **Combo**
  - 学校エコシステムシミュレーション（school_ecosystem_sim.py）  
  - 「1200時間問題」等の労働構造分析ドラフト  
  - メタ認知OS vs レガシーOS 記事案  
  - 電子黒板アプリを「あるべきDXサンプル」として提示  

### 5.4 Ritual Candidates

- **School Ecosystem Simulation Suite**  
  崩壊パターンをユーモア＋モデルで可視化する教材セット。

- **Teacher Time Cost Analyzer**  
  時間割・雑務・土曜勤務などを可視化するコスト分析ツール。

## 6. EX Spell：Electronic Blackboard OS

### 6.1 概要

- **Tier**: EX / Limit Break  
- 従来の「黒板・プリント・口頭説明」を置き換え、  
  授業の **UI・進行・ログ** を一元管理する **教室OS**。

### 6.2 役割

- 生徒：何を見て学べば良いかが明確になる。  
- 教員：授業構成・問いの流れをストック＆再利用できる。  
- DX視点：授業設計の「型」が可視化される。

### 6.3 合成要素

- UI/UX：1画面レイアウト・最小操作・シンプルな遷移  
- 教育デザイン：導入 → インプット → 演習 → メタ認知  
- 開発：GUI実装・ログ永続化・将来のLLM連携余地
